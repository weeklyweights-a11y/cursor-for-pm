# Phase 7: Evidence Briefs & Solution Design

> **Goal:** PMs can generate executive-ready evidence briefs for any theme with one click. Each brief is a structured document that answers: "Why should we build this?" with hard evidence. PMs can also describe a solution idea and the system evaluates it against the evidence, identifying gaps and strengths. Briefs are versioned, exportable, and shareable.
>
> **Done means:** A PM clicks "Generate Brief" on the #1 priority theme. Within 30 seconds, they have a polished document with: problem statement, customer impact analysis, evidence (quotes, segment breakdown, urgency distribution), business case, recommended action, and risks. They can edit it, regenerate sections, and export to markdown. They can also describe a proposed solution and get an AI evaluation: "This solution addresses 4 of 6 pain points. Here's what it misses."

## What You Are Building

Two things:

1. **Evidence Brief Generator** — Takes a theme and generates a structured, multi-section brief grounded entirely in the PM's data. Each section is generated by a focused LLM call with specific context. The PM can regenerate individual sections, edit inline, and export.

2. **Solution Evaluator** — The PM describes a proposed solution (free text). The system evaluates it against the theme's evidence: which pain points does it address? Which does it miss? What segments benefit? What's the predicted impact? This helps PMs validate their ideas before committing engineering resources.

The `sections` field stores an ordered array of brief sections:

Each section has:

- `edited` — True if PM has manually edited this section.
- `edit_history` — Array of previous versions (for undo). Store last 3 edits max.

## Brief Sections

Every evidence brief has these 7 sections, generated in order:

### 1. Problem Statement
**What it answers:** "What problem are customers experiencing?"

**LLM context:** Theme name, description, top 10 verbatim quotes, pain points from feedback items in this theme.

**Output:** 2-3 paragraphs summarizing the core problem. Written in third person ("Customers report that..."). Cites specific quotes.

### 2. Customer Impact
**What it answers:** "Who is affected and how badly?"

**LLM context:** Theme's segment_breakdown, unique_customers count, urgency_breakdown, list of affected customer names with segments.

**Output:** Breakdown by segment with specific customer names. Urgency distribution. How many customers, what percentage of the base. Written for executives who need to understand scope.

### 3. Evidence Summary
**What it answers:** "What's the evidence?"

**LLM context:** All verbatim quotes from feedback in this theme (up to 20), categorized by urgency. Topic and pain_point from each item.

**Output:** Organized evidence grouped by sub-theme or urgency level. Each group has 2-3 representative quotes. Not just a list — synthesized into a narrative.

### 4. Trend Analysis
**What it answers:** "Is this getting better or worse?"

**LLM context:** Feedback timestamps for items in this theme. Count by month/week. Any urgency escalation pattern (e.g., more "critical" items recently).

**Output:** Timeline analysis. "Mentions increased 40% in the last month." "3 of the 4 critical-urgency items were submitted in the last 2 weeks." If not enough historical data, say so honestly.

### 5. Business Case
**What it answers:** "Why should the business care?"

**LLM context:** Theme priority score and score_breakdown, PM's product goals, segment breakdown, unique customer count and their segments (with any revenue data if available in customer metadata).

**Output:** Connects the theme to business outcomes. "This theme affects 12 enterprise customers representing the highest-value segment." "Aligns with your stated goal of improving enterprise retention." References the priority score factors.

### 6. Recommended Action
**What it answers:** "What should we do?"

**LLM context:** Full theme data, PM's product context (existing features, known limitations), pain points, feature gaps extracted from feedback.

**Output:** Specific, actionable recommendation. Not "consider improving authentication" but "Implement SAML SSO integration as a first step, prioritizing Okta and Azure AD based on customer requests." Includes scope suggestion (what to include and what to defer).

### 7. Risks & Considerations
**What it answers:** "What could go wrong?"

**LLM context:** Theme data, sentiment breakdown, any conflicting feedback within the theme, related themes that might be affected.

**Output:** Honest assessment. "If we don't address this, X customers may churn based on urgency language." "Building this may pull resources from [other theme]." "Some feedback in this cluster may be about [related but different problem]."

## Generation Flow

### How Brief Generation Works

1. PM clicks "Generate Brief" on a theme (from Theme Dashboard or Theme Detail page).
2. API creates a brief record with status="generating".
3. A Celery task starts that generates each section sequentially:
   a. Load theme data (all stats, feedback items, quotes, scores).
   b. Load customer data for matched feedback items.
   c. Load PM's product context.
   d. For each section (1-7):
      - Build a focused prompt with the section-specific context.
      - Call the LLM.
      - Validate the response.
      - Store the section content in the brief's sections JSONB.
      - Update the brief record after each section (so the PM can see progress).
4. When all sections are done, set status="completed".
5. If any section fails, retry once. If retry fails, mark that section as "[Generation failed — click to retry]" and continue with remaining sections. Set brief status="completed" (partial is better than nothing).

### Why Sequential, Not Parallel?

Each section builds on a slightly different context, and running 7 LLM calls in parallel would strain the local Ollama setup. Sequential is simpler, more reliable, and the PM sees sections appearing one by one (better UX than waiting for all 7).

### Section Regeneration

PM can click "Regenerate" on any individual section. This:
- Saves the current content to edit_history.
- Calls the LLM with the same section-specific prompt.
- Replaces the content with the new generation.
- Sets edited=false, updates generated_at.

### Section Editing

PM can click "Edit" on any section to edit inline. This:
- Switches to an editable text area (markdown).
- On save: stores current content in edit_history, updates content, sets edited=true.
- Edited sections are NOT overwritten when PM regenerates other sections.

### Versioning

When PM generates a new brief for the same theme (e.g., after new feedback arrives and themes are re-clustered):
- The new brief gets version = previous version + 1.
- Previous brief's is_current is set to false.
- PM can view previous versions from the brief page.

## Solution Evaluator

### How It Works

1. PM opens a brief for a theme.
2. PM clicks "Evaluate Solution" (or a tab/section at the bottom of the brief).
3. PM types their proposed solution in a free-text area (e.g., "Implement SAML SSO with Okta and Azure AD. Add a settings page for admins to configure SSO. Auto-provision users from the identity provider.").
4. System sends the solution description + all theme data (pain points, feature gaps, customer impact, segments) to the LLM.
5. LLM evaluates: which pain points are addressed, which are missed, coverage score, strengths, gaps, recommended additions.
6. Result is stored in the brief's solution_evaluation field.
7. PM sees a visual evaluation: green checkmarks for addressed pain points, red X for gaps, coverage score as a percentage.

### Evaluation Prompt

**System prompt:** You are a product analyst evaluating a proposed solution against customer evidence. Be specific and honest. If the solution doesn't address a pain point, say so clearly.

**User prompt contents:**
- Theme name and description.
- All unique pain points from feedback items in this theme.
- All feature gaps from feedback items.
- Segment breakdown.
- The PM's proposed solution.
- Expected output schema (JSON with pain_points_addressed, coverage_score, strengths, gaps, etc.).

### Re-evaluation

PM can edit their solution description and re-evaluate. The previous evaluation is replaced (not versioned — evaluations are ephemeral, briefs are versioned).

## Non-Negotiable Rules for This Phase

Everything from Phases 1-6 still applies, plus:

1. **Briefs are grounded in data.** Every claim in a brief must trace back to actual feedback, theme stats, or customer data. The LLM must not fabricate quotes or stats.
2. **Sections generate sequentially, visible as they complete.** PM sees progress, not a 2-minute blank screen.
3. **Edited sections are preserved.** Regenerating other sections does not overwrite PM's manual edits.
4. **Edit history is stored.** PM can revert a section. Store last 3 versions.
5. **Previous versions are preserved.** When re-generating a full brief, old versions remain accessible. Don't delete.
6. **Solution evaluation is honest.** If a solution has gaps, say so. Don't rubber-stamp everything.
7. **Export produces clean markdown.** No JSON artifacts, no debug info. Clean, readable, shareable.
8. **Brief prompts in their own files.** 7 section prompts + 1 evaluation prompt, all in `backend/app/prompts/`.
9. **Brief generation never blocks the app.** Runs as Celery task. PM can navigate away and come back.
